3.1. Types and Type Systems

（类型与类型系统）

The notion of a type system can be difficult to define. For most programmers, the old adage, “I can’t tell you what it is, but I know it when I see it” describes their definition of a type system. Intuitively, we know that primitive types, classes, structs, and such are part of a type system, and that languages will enforce certain rules regarding the use of these types. But to actually say, in formal terms, what a type system is and entails is difficult. Nonetheless, most programmers, regardless of their background, will be able to infer some interesting details about the CLI type system from Example 3-1, even if they’re not familiar or comfortable with C#.

（类型系统的概念可能很难定义。 对于大多数程序员而言，“古谚曰：我没办法告诉你它是什么，但当我看到它时我就懂了”描述了他们对类型系统的定义。显而易见，我们知道原始类型、类、结构等是类型系统的一部分，并且语言将强制执行有关使用这些类型的某些规则。但实际上，很难正式的说类型系统是什么并且涉及什么。尽管如此，大多数程序员，无论其背景如何，都可以从示例 3-1 中推断出一些有关 CLI 类型系统的有趣细节，即使他们对 C# 并不熟悉或喜欢。）

Example 3-1. The Echo component revisited

Casual users of C, C++, or Java will find much here that feels familiar and intuitive. For example, the Echo component contains a number of type definitions: the enumerated type EchoVariation, the struct EchoValue, the interface IEchoer, and the class Echo. We can see some fields, some methods, some code and so. This is intuitive and familiar, even if we’ve never put formal definitions to it.

（普通的 C、C++ 或 Java 使用者会在这里发现很多熟悉和直观的内容。例如，Echo组件包含许多类型定义：枚举类型EchoVariation，结构EchoValue，接口IEchoer和类Echo。 我们可以看到一些字段，一些方法，一些代码等等。尽管我们从未对它进行正式定义，但其中许多都是直观且熟悉的。）

3.1.1. Type, Object, and Component

（类型，对象和组件）
…

A type is a specification that describes how a piece of unadorned data will be interpreted within the CLI execution engine. Types provide a way to classify both the shape of data and the ways that operations on that data should be expected to behave. The use of types has proven to be indispensable for the construction of reliable software on a large scale; their use results in a classification system that can be used to automate and enrich most aspects of the programming process, from compilation and linking to ensuring correct runtime behavior.

（类型是一种描述了如何在 CLI 执行引擎中解释未经修饰的数据的规范。类型提供了一种对数据的形状及应该对数据进行操作的方式进行分类的方法。实践表明，类型的使用对于构建大规模可靠软件是必不可少的。它们的使用产生了一个分类系统，该系统可用于自动化和丰富编程过程的大多数方面，从编译和链接到确保正确的运行时行为。）
…

3.1.2. Type Systems, More Formally Defined

The role of the type system in the CLI is twofold: it provides a logically consistent and unchanging conceptual foundation for programmers, and it ensures that programs can be checked for correctness at runtime. This latter role, enforcer of typesafety, helps to prevent tampering, and is an effective way to help ensure a robust, stable, and secure runtime environment. It is absolutely required when building services and applications that combine components from many sources.

（CLI 中类型系统的作用是双重的：它为程序员提供了逻辑上一致且不变的概念基础，并确保可以在运行时检查程序的正确性。后一个角色是类型安全的执行者，有助于防止篡改，并且是确保可靠，稳定和安全的运行时环境的有效方法。在构建多个来源的组件组合在一起的服务和应用程序时，这是绝对必需的。）

…

A system that checks (either statically or at runtime) to make sure that all types are being used correctly, as defined in their type descriptions, is said to be strongly typed. Strongly typed systems avoid erroneous or malicious computation by prohibiting operations that cannot be verified as typesafe; the ECMA specification contains a number of rules that define exactly what this means in the context of the CLI. When a component is JIT-compiled in the CLI’s execution engine, the compiler not only produces executable code, but also performs verification of that code using the rules specified in the ECMA specification. By ensuring that all loaded components are typesafe, the execution engine provides an important guarantee of integrity to component programmers.

（系统检查（通过静态或运行时）以确保正确使用所有类型（如其类型描述中所定义）的系统被称为强类型。强类型的系统通过禁止无法验证为类型安全的操作来避免错误或恶意的计算；ECMA 规范包含许多规则，这些规则准确定义了 CLI 上下文中的含义。在 CLI 的执行引擎中对组件进行 JIT 编译时，编译器不仅会生成可执行代码，而且还会使用 ECMA 规范中指定的规则执行对该代码的验证。通过确保所有加载的组件都是类型安全的，执行引擎为组件程序员提供了重要的可靠性保证。）

There are many benefits to using type systems :
（使用类型系统有很多好处：）

  • Detecting errors（错误检测）

    This is the most obvious advantage, by which types are used to detect areas of code in which the programmer has inadvertently asked for incorrect behavior taking a method call on a type that doesn’t exist, for example. It is better to detect this kind of error during development than at runtime, since you can rely on tools to do very complete checks.

    （这是最明显的优势，通过使用类型来检测程序员无意间要求错误行为的代码区域（例如，对不存在的类型进行方法调用）。与在运行时相比，最好在开发过程中检测这种错误，因为您可以依靠工具进行非常完整的检查。）

  • Maintenance （维护）

    This is an extension of error detection, in which programmers use typechecking as a powerful tool to support the refactoring of code. Instead of relying on programmer-centric disciplines when changing code to support new features or results, a programmer can simply change a type’s definition and run the compiler. The compiler will find the places where the code is no longer consistent, giving the programmer a well-defined and precise list of what needs to be changed to support the modification. While this technique might arguably fall into the category of slothful engineering practice, it is nonetheless very common.

    （这是错误检测的扩展，程序员在其中将类型检查用作支持代码重构的强大工具。更改代码以支持新功能或新结果时，不必依赖于程序员中心规则，而只需更改类型的定义并运行编译器即可。编译器将查找代码不再一致的位置，从而为程序员提供了良定义的精确列表，列出了需要更改以支持 modification 的内容。尽管可以说这种技术可能属于懒惰的工程实践范畴，但它却非常普遍。）

  • Abstraction （抽象）

    Strongly typed systems can enforce programmer discipline in ways that other tools simply cannot. This is particularly true within object-oriented systems that support inheritance (either interface-based or implementation-based) then a method expects a parameter of type Person, for example, the typechecker enforces that only Persons, and derived types, will be accepted. This in turn gives the programmer a powerful means to differentiate between Persons and other types, such as Lists, Forms, and XmlReaders, making code clearer and more intentional.

    （强类型的系统可以以其他工具根本无法执行的方式强制执行程序员规则。在支持继承（基于接口或基于实现）的面向对象系统中，这尤其如此。例如，当一种方法期望参数类型为Person时，类型检查器将强制只将Person和派生类型作为参数。反过来，这又为程序员提供了一种强有力的手段来区分Person和其他类型，例如Lists、Forms和XmlReaders，从而使代码更清晰，更有针对性。）

  • Documentation （文档）

    Types are also handy when reading programs, since the structure of type declarations helps offer hints regarding their usage and behavior. This sort of documentation is especially useful because, unlike comments, there is no way for it to become outdated or inaccurate.

    （在读取程序时，类型也很方便，因为类型声明的结构有助于提供有关其用法和行为的提示。这种文档特别有用，因为与注释不同，它不会变得过期或不准确。）

  • Efficiency （效率）

    If the type of an argument can be extracted at runtime, then optimizations can be made on that type to increase program speed, reduce memory footprints, or both.

    （如果可以在运行时提取参数的类型，则可以对该类型进行优化以提高程序速度，减少内存占用，或两者兼而有之。）

  • Security （安全）

    A typechecker can enforce a policy that says that types are not to be used in ways which would allow for malicious code to subvert the program or act in other undesirable ways. Some languages may choose to allow programmers to override a strict policy for the purposes of interfacing with code that is not typesafe; in this case, it is important to permit these programmers to make explicit assertions about their intentions, prove that they are authorized to make such assertions, and then include these assertions in the type-checking process.

    （类型检查器可以强制执行一项策略，规定不得以允许恶意代码颠覆程序或以其他不良方式采取行动的方式使用类型。某些语言可能选择允许程序员重写严格的策略，以与非类型安全的代码进行交互。在这种情况下，重要的是允许这些程序员针对其意图进行明确的声明，证明他们有权进行此类声明，然后将这些声明包括在类型检查过程中。）

A strongly typed system can offer all of these benefits, without inconveniencing the programmer in the least.
（强类型系统可以提供所有这些好处，而不会给程序员带来任何一点的麻烦。）
